Coach Platform — architecture explained

A visual walkthrough of the codebase: system architecture, DB schemas, the OpenClaw-style memory engine, the Ask/Plan/Auto modes, the Garmin sync routine, and the next-week build flow. All file paths reference coach-platform/api/src unless noted.

1. System architecture

Three deployables and five external dependencies. The API owns everything: MongoDB state, the agent layer, Redis locks, LLM calls. The fetch service (coach-platform/fetch-service) is deliberately dumb — credentials in, metrics out, no database of its own.

2. Database schemas

There are 12 collections. First the core training data — everything hangs off user_id, and the one real FK chain is training_profiles → programs → planned_sessions.

Design decisions worth noticing:

The plan/reality split is the heart of the model. planned_sessions is what the coach intends; sessions is what Garmin says actually happened. The outcome object on a planned session (status completed / skipped / deviated, plus matched_activity_id) is the bridge — SessionMatcherService fills it in after every sync. programs embeds its weeks as subdocuments, and each week carries week_state (open → targets_locked → locked) and a weekly_targets object — that lock state powers the "domino effect" fix. Every ingested document (sessions, recovery_daily, performance_daily) carries a content_hash, and plan_state on sessions is either tentative or committed — the entire HITL system rides on that one enum. A partial index on programs ({user_id} unique where status='active') enforces one active program per user at the DB level, not in application code.

3. The OpenClaw-style memory engine

The pattern is "durable log as truth, rebuildable index as convenience." In OpenClaw the log is files and the index is search; here the log is preference_events and the index is the user_preferences projection. Nothing ever mutates the log — every "learning" the system does flows through this pipeline:

What each stage does

The log (personalization/domain/preference-event.model.ts). Every signal — "I hate burpees", a skipped session, a session-end flush — becomes one immutable event. The key design choice is that confidence (explicit vs inferred) is a first-class field written at capture time, so "the user said it" and "the system guessed it" never get mixed. raw_text keeps the verbatim phrasing for audit, and taxonomy_version (currently 5) lets the tag vocabulary evolve without corrupting old events — that's how the overreaching safety tag landed as a clean v4→v5 bump.

Distillation (DistillationService.distill()). A pure function: it filters standing events for the discipline, groups them by logical bucket (avoid-lists, biases, setpoints, prescriptions), and builds entries. Same events + same now always yield the same projection — determinism comes from sorting by event date, never by insertion order. That purity is exactly what makes the rebuild endpoint possible: RebuildProjectionHandler fetches everything via findAllForReplay(userId), re-distills, and upserts. Delete the projection, hit rebuild, and it comes back identical.

Promotion (PromotionService.buildEntry) is the threshold state machine:

SignalRuleResult
explicit + standingimmediatehard preference
inferred dislike3+ occurrences within 90 dayssoft preference
inferred like2+ occurrences within 90 dayssoft preference
inferred, under threshold—not materialized yet
inferred, last reinforced >90 days ago—decayed, pruned
inferred later stated explicitly—hard + confirmed: true

The asymmetry (3 for dislike, 2 for like) is deliberate — the cost of wrongly avoiding an exercise is higher than wrongly preferring one, so dislikes need more evidence.

The validator (ProjectionValidatorService) is a guardrail on memory itself, re-checked after every distill: no inferred entry sitting at hard (demoted), no decayed entry left un-pruned (dropped), biases clamped to ±1.0, no entry without sourceEventIds provenance (dropped), supportCount must equal the evidence list length (reset), no duplicates (deduped). Breaches are repaired before persistence and surfaced for audit.

The read path (prompt-flattener.ts) is what makes any of this matter — the projection is rendered into plain [must] / [prefer] / (inferred) text blocks seeded into every agent's prompt. Cold start: a projection with sourceEventCount: 0 signals a brand-new user, so agents fall back to the onboarding survey + wearable baseline instead of an empty preference block.

One layer below, the same delta-sync idea applies to raw physiology: common/util/content-hash.ts computes SHA-256 over sorted-key JSON, so re-polling Garmin for an unchanged day produces the same hash and writes nothing — zero DB churn, zero downstream token spend.

4. The engine every agent runs on: the agentic loop

Every agent — Coach, Recovery, Planner, Assistant — is the same bounded loop (agents/shared/llm/agentic-loop.runtime.ts):

Three things make this loop safe and cheap. First, the exit condition is structural: the loop only ends when the model calls a tool marked terminal: true (e.g. commitPlacementTool), so "done" is a typed, validated payload — never free text parsed by hope. Second, the validator bounce: a failed Zod parse or a thrown handler error doesn't crash the run; it's fed back as a tool-result message ("Invalid arguments…") so the model self-corrects within the 8-turn budget. Third, if the budget exhausts without a terminal call, the loop returns exhausted: true — and the orchestrator's requireTerminal() then throws, aborting the whole pipeline. Since every stage writes tentative-only, an aborted run commits nothing user-visible.

5. The three modes — flow, tools, permissions

Modes live on the conversation (conversations.mode), and the crucial point is that all three run identical agent code — the mode only changes which write paths are reachable in assistant.decision.ts. The Assistant LLM classifies each turn into a lane: white (pure question), black (explicit order), gray (soft signal, needs one clarifying question). Then the mode gate applies:

Mode by mode

Ask is a structural read-only gate, not a polite prompt instruction. In assistant.decision.ts: if the lane isn't white or a week edit is present, the decision returns intentBlocked: true with a "switch to Plan mode" hint — nothing is captured, no pipeline fires, regardless of what the model outputs. You can still consult all 12 read tools for trends and stats.

Plan is where everything writeable happens, but two guards sit before any fire: a non-safety tag only triggers a pipeline if affectsCurrentWeek: true (otherwise it's just logged as a preference for future weeks — the WRITE_ONLY path), and an unconfirmed week edit that breaches locked targets sets a clarifyingQuestion and blocks until you confirm. The result of any fired pipeline is always tentative + an approval card.

Auto routes the turn to AutoModeOrchestratorService, which runs a Coach↔Recovery debate graph, applies its own guardrails (target swing, week volume, readiness degradation), and flips tentative→committed with no card. Every run is recorded as an AutoModeRun with full trace and diffs, so autonomy is auditable and revertible after the fact.

Two permission principles run through this table. Tenant scoping: every tool handler gets ctx.userId from the runtime context, and every CQRS query filters on it — the model can never pass a user id as an argument. 6. The routine flow — what a Garmin sync triggers

This is the "magic moment" chain, and it's fully deterministic until the pipeline itself runs:

The cron (agents/triggers/garmin-sync.scheduler.ts) sweeps every minute but fires per user only when one of their configured local sync times (timezone-aware, up to 3/day) comes due — and the slot is claimed atomically, so two app instances can't double-fire. IngestionOrchestrator.runForUser (ingestion/application/ingestion.orchestrator.ts) decrypts the Garmin credentials via CryptoService (AES-256-GCM, key from a secrets manager) and calls the fetch service in 5-day chunks; each returned day is routed to UpsertRecoveryDay / UpsertPerformanceDay / UpsertSession commands, which compare content_hash and return {written: false} on an unchanged day.

The significance gate (evaluateSyncSignificance) is the part that makes Auto mode feel proactive without being noisy: only a new day with a missed session, unplanned load, or adherence deviation escalates. If it does, FetchTrigger enqueues FULL_SESSION_DAY with a garmin-sync:{userId}:{date} run id — Redis SET-NX (24h TTL) kills duplicate runs, a per-user lock plus the in-process promise chain in pipeline-queue.service.ts guarantees a scheduled sync and a simultaneous mid-chat edit never race, and a later run for the same week marks the earlier result superseded so stale approval cards die. The pipeline then runs Recovery → Coach → Planner: that's how a low-HRV morning becomes an already-downgraded session before you open the app. (AutoModeReaperService sweeps every 5 minutes for crashed auto runs.)

7. Building the next week

The build is a fixed stage sequence where every stage is paired with a code-level guardrail — this pairing is the answer to the "domino effect" problem:

The ordering is the whole trick. Targets get locked (LockWeeklyTargetsCommand, week flips to targets_locked) before a single session exists. From then on, every session the Coach drafts passes validateAgainstWeeklyTargets(proposed, committedSoFar, targets) — cumulative count and volume against a frozen budget. So "does this edit break the week?" stops being a reasoning problem and becomes arithmetic the code runs every time. A violation doesn't crash anything: it throws inside the tool handler, bounces back into the agentic loop as an error message, and the model redrafts within its 8-turn budget. validateWeek() (agents/coach/coach.guardrails.ts) adds the physiological caps: ≤+10% week-over-week load (waived on deloads), red readiness → 0 hard sessions, amber → 1, and every session must carry coachNotes. On the Planner side, validatePlacement() rejects overlaps, blocked-window hits, bad tz→UTC conversion, and doesn't-fit durations before the only irreversible external write — and even that write goes through assertOwned(), which refuses to touch any calendar event not tagged with the app's own extendedProperties.private.appId.

There are two ways a week gets built. The batch path (what the diagram shows) runs all stages back-to-back and lands as one approval-card batch: per-session cards with the coach rationale, planner placement note, and a diff vs the committed baseline (new / modified / unchanged / removed, keyed by slot_key). Approve flips tentative→committed and syncs the calendar; reject discards the draft and falls back to the previous committed week — and approval.policy.ts only offers reject when a committed fallback actually exists, so a first-ever week can't be rejected into nothingness. The interactive path (agents/build/build-conversation.orchestrator.ts) is the Q&A loop: the Coach first proposes targets conversationally, then drafts one session at a time, the Planner offers 3 candidate slots per session and you pick — approval happens inline per session, so there's no separate card batch at the end.
